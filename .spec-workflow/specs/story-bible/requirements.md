# Requirements — Story Bible ("The Tome Plans Ahead")

Status: DRAFT FOR REVIEW (2026-07-10). Companion docs: `design.md`
(architecture), `tasks.md` (agent tasks).

Grounding: this spec operationalizes product principles 3–4
(`steering/product.md` — "a flavor-only choice is an anti-feature",
"consequences are immediate and felt") and closes the dead-key defect class
surfaced in the 2026-07-10 PM/UX panel review: locked `has_item` choices whose
key no reachable path ever grants. It extends story-engagement R1 (story arc),
R4 (locked choices), R10.3 ("items are keys") and implements the
specced-but-unbuilt `choice.locked_shown` event (story-engagement R16.1).
Where this doc conflicts with older specs, THIS DOC WINS for the story-bible
scope. Story-engagement's cross-cutting rules (R16.2 safety, R16.3
back-compat, R16.4 testing) apply to every requirement below unchanged.

## Introduction

Turn 1 already plans the story's SHAPE: the model emits a `storyArc`
(dramatic question, beats, candidate endings) alongside the opening scene
(story-engagement R1.1). It never plans the story's NOUNS — no item exists
before the scene that names it, so every locked `has_item` gate is an id
invented in that same breath, with no obligation ever to grant it. The Story
Bible is a second, larger planning artifact generated by a dedicated LLM call
that runs in the background at save creation, racing turn 1 behind the
opening cinematic: a key registry, a lock plan, a cast sheet, a twist bank,
and ending requirements. Per-turn prompts then gate only against planned or
already-granted keys; a hallucinated gate becomes a registry admission (a
promise the engine keeps via the existing delayed-thread machinery) instead
of a dead end. The reader keeps full freedom — the bible is gravity, not
rails: assets are pinned to conditions and bands, not to scenes, so they
survive any free-text detour.

## Alignment with Product Vision

- **Agency is the point** (principle 3): a door that can never open is a
  flavor-only choice wearing a lock. Registry-enforced gates make every lock
  a real, pursuable goal.
- **Consequences are immediate and felt** (principle 4): a teased key that
  later arrives — narrated, as a fired thread — is the "story remembering"
  loop at its best.
- **Story first** (principle 1): the bible is server-only scaffolding; the
  reader only ever feels a world that seems to have existed before they
  arrived.

## Requirements

### R1 — The Bible Call (dedicated background generation at save creation)

User story: as a reader, the world I explore was planned before I stepped
into it — keys exist before their doors.

1. WHEN an llm-driven save is created THEN the server SHALL schedule a
   dedicated background LLM action (the "bible call") seeded from the same
   premise/seed inputs as turn 1. Turn-1 scene generation SHALL NOT wait on
   it, and the bible call SHALL NOT delay any reader-facing response (it
   races turn 1 behind the opening cinematic loader).
2. The bible call SHALL request a structured `storyBible` object:
   - `keyRegistry`: 6–12 entries `{id (slug ≤48), label ≤80, opensHint ≤120,
     surfaceBand: early|mid|late}` — the only items the story plans to gate on.
   - `lockPlan`: 2–5 entries `{id (slug ≤48), label ≤80, keyId (MUST
     reference a keyRegistry id), gateBand: mid|late, note ≤120}` — planned
     doors, each with a key that surfaces in an earlier-or-equal band.
   - `cast`: 2–5 entries `{id (slug ≤48), label ≤80, want ≤120, secret ≤120,
     bondHint ≤120}`.
   - `twists`: 2–4 entries `{id (slug ≤48), label ≤80, precondition ≤120}`.
   - `endingHints`: 2–4 entries `{endingId (slug ≤48), requires ≤160}` —
     requirement sketches for likely endings.
   - `motifs`: 3–6 strings ≤40 — imagery/tone anchors.
3. A pure `validateProposedBible` (same clamp/slug/dedupe discipline as
   `validateProposedArc`, `packages/engine/src/arc.ts:114`) SHALL clamp every
   string, sluggify + dedupe ids, drop `lockPlan` entries whose `keyId` does
   not resolve to a registry entry, and drop `lockPlan` entries whose
   `gateBand` precedes their key's `surfaceBand`. A bible salvageable to ≥4
   registry keys is kept; anything less is discarded whole.
4. WHEN the bible call fails, times out, or returns an unsalvageable payload
   THEN the save SHALL play bible-less forever under exactly today's behavior
   — never a turn failure, at most ONE retry (BC5 spirit). Bible-less saves
   remain a permanently supported path (BC9).
5. WHEN a turn completes and a validated bible exists for the save but is not
   yet attached THEN the engine SHALL attach it: `endingHints` are
   fuzzy-matched to the turn-1 arc's `candidateEndings` ids (reusing
   `normalizeEndingId` semantics); unmatched hints are dropped. Attachment on
   any turn ≥2 is acceptable; there is no attach deadline.
6. Exactly one bible call per save, plus at most one refresh call per act
   boundary (R6). No bible calls for authored (non-llm) stories, co-op
   follower clients, or forks that copy an existing bible.

### R2 — Storage, spoilers, safety

1. The bible SHALL be stored in a dedicated table keyed by `saveId`
   (integrator-owned schema change, BC7) — NOT inside `saves.state` (it is
   large and mostly static). Per-entry consumption state (`granted`,
   `opened`, `fired`, `retired`) SHALL be updated in the same mutation that
   applies the turn, so bible state can never disagree with save state.
2. NO field of the bible SHALL ever reach the client. `projectLlmDrivenScene`
   remains the single choke point (BC10); a projection test SHALL assert the
   absence of every bible field, matching the existing spoiler-test pattern.
3. Every bible string SHALL pass `evaluateTextPolicy` before persistence
   (story-engagement R16.2); blocked strings are neutralized, never a
   failure.
4. Forking a save SHALL copy the bible row (with consumption state as of the
   fork point) so forks inherit the same planned world.

### R3 — Prompt integration: the digest (gravity, not rails)

User story: as a reader I can type anything — the story bends toward its
plans without ever railroading me.

1. WHEN a scene prompt is built for a save with an attached bible THEN it
   SHALL carry a STORY BIBLE digest section containing ONLY: unconsumed
   `keyRegistry` entries whose `surfaceBand` is due (band ≤ current turn
   band, reusing the `beatBandForTurn` bands), `lockPlan` doors due or
   promised, the cast sheet, and due twists. The full bible is NEVER inlined.
2. The digest SHALL instruct opportunistic placement: surface due assets
   when the fiction makes it natural; the reader's actions always win; a due
   asset may be relocated or delayed, never forced. (Mirrors the arc
   steering language of story-engagement R1.3.)
3. The GATED CHOICE rule SHALL be tightened: `has_item` conditions may
   reference ONLY (a) ids in the digest's key registry, (b) items currently
   held, or (c) items granted earlier this run. Teasing a brand-new key
   means granting it this scene or naming it as a registry key first.
4. Token budget: digest + rule edits ≤600 tokens over the current baseline
   on any single prompt, measured in the existing prompt snapshot test
   (extends story-engagement R16.5's discipline).
5. Saves without a bible (legacy, failed call, authored) SHALL render the
   prompt exactly as today — the digest section and rule tightening are
   conditional on attachment (BC9).

### R4 — Registry-enforced gates: dead keys impossible by construction

User story: every locked door I see has a key that exists somewhere ahead.

1. A new optional `itemsEverGranted` ledger (normalized ids via
   `normalizeItemRef`) SHALL be appended on every `inventory_add` on the llm
   path. Optional field read with `?? []` — NO migration rewrite
   (story-engagement R16.3).
2. WHEN post-processing a proposed scene, every locked `has_item` choice
   SHALL be resolved against, in order: current inventory + `itemsEverGranted`
   (tolerant matching), a sibling choice in the same proposal that grants the
   item, then the bible `keyRegistry`. A registry match marks that key
   `promised` (recording the turn) and keeps the choice locked.
3. WHEN the gated id matches nothing anywhere THEN: if the registry has
   capacity (registry size under 16) the id SHALL be ADOPTED into the registry as
   a floating key (`surfaceBand` = current band) and treated as promised;
   otherwise the choice is auto-unlocked via the existing
   `unlockChoiceResult` path. At most 2 keys may be `promised`
   simultaneously; a proposal that would exceed the cap has its excess gates
   auto-unlocked.
4. The ≤1-locked-per-scene keep-rule SHALL prefer the most ATTAINABLE lock
   (registry-promised or smallest stat/currency deficit) rather than the
   first by array order.
5. Bible-less saves get R4.1–R4.4 with an empty registry: never-granted,
   never-adoptable gates auto-unlock (this alone kills phantom locks on
   legacy saves).
6. Analytics (fire-and-forget, `analytics_events` pattern):
   `bible.generated`, `bible.attached`, `bible.refreshed`,
   `bible.key_promised`, `bible.key_adopted`, `bible.key_seeded`,
   `bible.gate_phantom_unlocked`, and — implementing story-engagement R16.1 —
   `choice.locked_shown` with payload `{conditionKind, itemId?, everGranted,
   inRegistry, deficit?}`, fired once per turn at visibility computation
   (NOT per render).

### R5 — Promise keeping: the engine seeds the key

User story: a door the story teased me with becomes a hunt that pays off.

1. WHEN a promised key remains ungranted 3 completed turns after its promise
   THEN the engine SHALL seed it deterministically by scheduling a delayed
   thread (`delayNodes: 1`, effect: the `inventory_add`, note derived from
   the key's `label`/`opensHint`) through the existing thread machinery
   (story-engagement R3) — the key arrives narrated as a fired-thread
   callback, indistinguishable from authored foreshadowing.
2. WHILE any key is promised THEN the prompt SHALL carry an OUTSTANDING KEYS
   line (inside the digest budget): surface this key naturally within its
   band. WHEN a promised key is granted THEN the line flips to instruct
   re-offering the locked door within the next 2 scenes; the promise is
   consumed when a matching gate is next proposed or the door is narrated
   open.
3. No new client UI in v1: the seeded key rides the existing thread badge and
   effect pips. (A reader-facing "doors journal" is explicitly deferred —
   see Non-Goals.)

### R6 — Act-boundary refresh (the plan survives contact with the reader)

1. WHEN the act advances (`advanceActIfDue`) on a bible save THEN the server
   SHALL schedule ONE background refresh call carrying the current bible,
   consumption state, and the rolling story summary; the model may relocate
   or retire unconsumed entries and add ≤2 new registry keys. Output passes
   `validateProposedBible` merge rules; consumed/granted entries are
   immutable; a failed refresh keeps the old bible silently.
2. The refresh SHALL never un-promise a promised key or remove a key that a
   pending lock references.

## Non-Goals (explicit)

- **Speculative scene pre-render** (generating scene N+1 while reading scene
  N) — independent latency experiment, separate spec if pursued.
- **Reader-facing doors journal / bible UI** — candidate follow-up once
  telemetry shows the loop working.
- **Cinematic pre-warm from bible motifs/ending imagery** — coordination
  point for omni-cinematics; the bible schema reserves `motifs` for it, but
  no media work ships in this spec.
- **Authored-story reachability linting** — related but separate workstream
  (creator/publish pipeline).

## Non-Functional Requirements

### Performance
- Turn-1 latency unchanged (bible call is fully asynchronous).
- Attachment/post-processing adds only in-memory work to the turn mutation —
  no extra LLM call on any reader-facing path.
- Cost envelope: ≤3 extra LLM calls per save lifetime (1 bible + ≤2 act
  refreshes), each smaller than one scene call.

### Reliability
- Every bible feature degrades to today's behavior: no bible, no digest, no
  registry enforcement beyond R4.5's ledger check. The bible is never a
  turn-failure source (BC5 discipline end to end).

### Code Architecture
- All new engine logic pure + table-driven-tested (BC6, R16.4); convex
  handlers get fake-ctx tests; schema/crons/index changes are
  integrator-owned (BC7); `exactOptionalPropertyTypes` conventions (BC4).

### Security / Safety
- Bible content passes the same text-policy gate as arc/thread text (R2.3);
  bible rows are scoped by save ownership like every per-save table.
